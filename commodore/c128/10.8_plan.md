# 10.8 — C128 Preload Cache Rebaseline Plan (With Required Bank 1 Ownership Refactor)

## Why 10.8 failed twice

Both attempts made the same category error:

1. They treated Bank 1 as "unused RAM" without proving what still lives there after `boot128`.
2. They introduced preload/cache behavior before establishing a valid Bank 1 ownership model.
3. They mixed loader semantics, cache semantics, overlay policy, and MMU hardening in one effort.

The immediate crash signature proved the underlying issue:

- `boot128` loads the full program image into Bank 1, then copies it into Bank 0.
- The staged Bank 1 image remains resident after boot.
- A cache region placed at overlapping Bank 1 addresses is not data-only scratch space.
- Any MMU leak can execute cached payload bytes as code.

Therefore, **Bank 1 ownership refactor is not optional follow-up work. It is Phase 0 of 10.8.**

---

## Core decision

10.8 must now be defined as:

1. **Bank 1 ownership proof and refactor**
2. **MMU gateway hardening for Bank 1 access**
3. **Only then** tier-only preload cache
4. **Only after that** overlay-cache design and implementation

If Phase 0 is not complete, preload cache must remain disabled.

---

## Non-negotiable invariants

### Memory ownership invariants

1. Every Bank 1 range used by the C128 build must have one owner and one purpose.
2. No preload/cache bytes may be stored in any Bank 1 address that aliases executable or control-sensitive Bank 0 code/data.
3. Cache storage must live in a Bank 1 region proven safe by boot-time ownership analysis, not by convenience.
4. Bank 1 usage must be documented in one place and compile-time asserted.

### MMU invariants

1. Runtime must resume in Bank 0 all-RAM mode after every Bank 1 helper.
2. No gameplay control flow may continue under ambient Bank 1 selection.
3. Direct ad-hoc `sta $ff00` in mixed gameplay/load paths is forbidden unless the routine is a documented MMU gateway/helper.
4. All Bank 1 accesses used by preload/cache must go through centralized helpers that restore Bank 0 before return.

### Cache invariants

1. C128 preload cache is not REU emulation.
2. Cache control metadata lives in Bank 0 only.
3. Tier cache and overlay cache are separate readiness domains.
4. Overlay cache remains disabled until tier-only cache is stable after the memory-ownership refactor.

---

## Phase 10.8.0 — Bank 1 Ownership Refactor (Required Prerequisite)

### Goal

Establish a valid Bank 1 memory map and remove the false assumption that Bank 1 contains free scratch space after boot.

### Deliverables

1. Document the post-boot Bank 1 ownership model:
   - common/shared RAM
   - map region
   - tier DB mirror region
   - staged Bank 1 program image copied by `boot128`
   - any truly reclaimable region
2. Prove, with exact address ranges, whether any Bank 1 cache window exists on the current 128KB design.
3. Add compile-time assertions for all Bank 1 ownership boundaries.
4. Update architecture docs and comments so no file describes Bank 1 preload space as "unused" without proof.

### Required analysis work

1. Trace `boot128` load/copy behavior and define the post-copy residual Bank 1 image footprint.
2. Compare that footprint against:
   - map storage
   - tier DB staging region
   - any planned preload/cache window
3. Decide one of the following:
   - **A. No safe preload window exists on current layout** → keep preload disabled and re-scope 10.8 to architecture change first.
   - **B. A safe preload window does exist** → document exact start/end addresses and assert them.

### Acceptance criteria

1. A single source of truth exists for Bank 1 ownership.
2. There are compile-time asserts preventing overlap between:
   - map
   - tier DB mirror
   - staged program image
   - preload cache window
3. No code or comments still describe overlapping staged-image space as safe cache space.
4. If no safe window exists, the code explicitly keeps C128 cache disabled.

## Phase 10.8.1 — MMU Gateway Refactor

### Goal

Make Bank 1 access structurally safe before reintroducing preload/cache behavior.

### Deliverables

1. Inventory all C128 paths that flip `$FF00` directly.
2. Classify each as:
   - allowed MMU gateway/helper
   - legacy direct use to be replaced
3. Centralize preload/cache Bank 1 access through low-memory helpers only.
4. Ensure every helper:
   - enters Bank 1 locally
   - performs the read/write/copy
   - restores Bank 0 all-RAM
   - returns with documented clobbers

### Rules

1. No preload or cache call site may manually manage mixed MMU state.
2. No higher-level loader routine may branch or tail-call while still in Bank 1 context.
3. Interrupt/vector hardening is not a substitute for ownership correctness and must not be used to justify unsafe cache placement.

### Acceptance criteria

1. Bank 1 access sites are auditable and centralized.
2. Existing tests pass after direct MMU cleanup.
3. Additional targeted assertions/tests cover helper restore behavior where practical.

---

## Phase 10.8.2 — C128 Cache Control Plane

### Goal

Separate C128 preload/cache state from REU state before enabling any cache path.

### Deliverables

Introduce Bank 0 cache state only:

- `c128_cache_enabled`
- `c128_cache_tiers_ready`
- `c128_cache_overlays_ready`
- `c128_cache_failed`
- per-tier ready bits
- per-overlay ready bits
- cache size/offset tables if and only if a safe Bank 1 window exists

### Rules

1. `reu_present` means hardware REU only.
2. C128 title/sysinfo and startup preload policy must not key off fake REU state.
3. If Phase 10.8.0 did not prove a safe cache window, `c128_cache_enabled` must remain off by default and no preload path may run.

### Acceptance criteria

1. C128 and C64 semantics are clearly separated.
2. No fake REU advertisement exists on C128.
3. If cache is disabled due to unsafe ownership, runtime remains stable on disk-only flow.

---

## Phase 10.8.3 — Shared Preload Transaction Layer

### Goal

Create one preload-only file-I/O transaction path before any cache behavior is enabled.

### Required transaction sequence

1. save any caller state needed
2. `SEI`
3. enter known KERNAL state:
   - `$FF00 = MMU_NORMAL`
   - `$01 = BANK_ALL_ROM`
4. pre-close logical file `2`
5. `CLRCHN`
6. `SETNAM`
7. `SETLFS`
8. `SETBNK`
9. `LOAD`
10. capture status
11. `CLOSE`
12. `CLRCHN`
13. restore runtime state:
   - `$FF00 = MMU_ALL_RAM`
   - `$01 = runtime default`
14. reassert VDC mode if needed
15. restore interrupt state
16. return explicit success/failure

### Rules

1. Startup preload callers must use only this transaction path.
2. Runtime disk loads may remain separate until later.
3. This phase does not enable overlay cache.

### Acceptance criteria

1. Existing tests pass.
2. Repeated preload loads do not leak file handles.
3. The transaction path is not used as evidence that cache storage is safe; that was decided in Phase 10.8.0.

---

## Phase 10.8.4 — Tier Cache Only (Only If Phase 10.8.0 Proved Safe Storage)

### Goal

Enable tier-only cache in a Bank 1 region proven safe by the ownership refactor.

### Preconditions

All of the following must already be true:

1. A non-overlapping Bank 1 cache window is documented and asserted.
2. MMU access has been gatewayed/centralized.
3. C128 cache control state is independent of REU.
4. Shared preload transaction path is in place.

If any precondition is false, this phase must not be implemented.

### Deliverables

1. Startup preloads `MONSTER.DB` files only.
2. Overlay preload remains disabled.
3. Tier cache uses:
   - transaction load to `$E000`
   - copy to safe Bank 1 cache slot
   - byte verification at preload time
   - per-tier ready-bit enable only after verify succeeds
4. Runtime tier fetch:
   - checks only tier cache readiness/bit
   - copies cached payload to `$E000`
   - falls back to disk on miss/failure

### Acceptance criteria

1. New game, town, first dungeon entry, and stair transitions work without JAM.
2. Missing one cached tier falls back only for that tier.
3. No overlay cache code is active.

## Phase 10.8.5 — Smoke Coverage Upgrade

### Goal

Test the real risk boundary instead of `entry_main`.

### Required tests

1. `boot_title_newgame_smoke`
   - boot D64
   - survive preload
   - reach title input loop
   - inject `N`
   - reach `game_new_start`
   - continue cycles after transition
2. `boot_tier_transition_smoke`
   - start new game
   - reach dungeon
   - cross at least one tier boundary
   - continue execution after transition
3. `preload_partial_failure_smoke`
   - use a test build that skips preloading one tier while keeping the disk asset present
   - confirm runtime falls back through `tier_load_disk`
   - confirm execution continues after the fallback path engages

### Rules

1. These tests must exist before overlay-cache work begins.
2. They must be considered gatekeepers for re-enabling any C128 cache feature.

## Phase 10.8.6 — Overlay Cache Design Lock

### Goal

Define overlay cache only after the memory-ownership and tier-cache work are proven stable.

### Constraints

1. Overlay cache storage must also live in a Bank 1 region proven safe by Phase 10.8.0.
2. Overlay cache must not share readiness semantics with tier cache.
3. Overlay fetch must not use per-fetch byte-for-byte runtime verification loops.
4. Overlay preload must not be enabled until the tier-only path and upgraded smokes are stable.

### Locked design decisions

1. Overlay cache uses four fixed 4 KB Bank 1 slots, one per overlay:
   - `OVL_STARTUP`  -> `$A000-$AFFF`
   - `OVL_TOWN`     -> `$B000-$BFFF`
   - `OVL_DEATH`    -> `$C000-$CFFF`
   - `OVL_DUNGEON_GEN` -> `$E000-$EFFF`
2. `$A000-$DFFF` is chosen deliberately instead of packing overlays directly after the tier-cache tail:
   - it is fully inside the reclaimed high Bank 1 region
   - it is 4 KB aligned for exact overlay copies
   - it avoids introducing variable-size slot arithmetic into the runtime fetch path
   - it avoids `$D000-$DFFF`, where standard Bank 1 helper mode still exposes I/O instead of RAM
3. Overlay cache metadata stays in Bank 0 only:
   - `c128_cache_overlays_ready`
   - per-overlay ready bits
   - fixed slot table
   - optional preload status/debug byte
4. Tier and overlay readiness remain independent:
   - tier cache may be ready while overlay cache is disabled
   - overlay cache failure must not clear tier cache readiness
5. Overlay preload uses the shared preload transaction path and the existing `$E000` overlay load contract:
   - preload overlay PRG to `$E000`
   - copy exactly `$1000` bytes into its Bank 1 slot
   - verify bytes once during preload
   - set ready bit only after verification succeeds
6. Runtime overlay fetch is copy-only:
   - copy exactly `$1000` bytes from Bank 1 slot to `$E000`
   - set `current_overlay`
   - do not run full-byte runtime verification loops on every fetch
7. Runtime failure policy is class-local:
   - if preload of an overlay fails, leave overlay cache disabled and keep disk overlay loading active
   - if runtime overlay cache fetch fails, disable only overlay cache for the session and retry via disk
8. Overlay cache must preserve the current Gate A tier behavior:
   - cached tiers remain valid across overlay loads
   - uncached tier/name state must still be invalidated when overlays overwrite `$E000`

### Required compile-time guards

1. Assert each overlay slot stays within the reclaimed high Bank 1 region.
2. Assert overlay slots do not overlap the tier-cache window `$8000-$94F7`.
3. Assert overlay slots do not overlap any reserved/common Bank 1 area.
4. Keep overlay slots fixed even if future free-space math would allow denser packing; the fetch path depends on this simplicity.

### Acceptance criteria

1. Overlay-cache storage is proven non-overlapping.
2. Overlay-cache implementation plan is decision-complete before code lands.

---

## Phase 10.8.7 — Overlay Cache Only (Gate B)

### Goal

Enable overlay preload/fetch only after every prerequisite above is satisfied.

### Deliverables

1. Startup preload loads overlays after tiers.
2. Each overlay gets a fixed slot in proven-safe Bank 1 storage.
3. Runtime overlay fetch uses cache if ready, otherwise disk.
4. Failure disables only overlay cache for the session.

### Acceptance criteria

1. Town/store/death/dungeon-gen overlay transitions are stable.
2. No JAM in repeated title/new/dungeon/death/town cycles.
3. Dedicated `boot_title_idle_smoke`, `town_overlay_smoke`, `town_overlay_female_smoke`, and `death_overlay_smoke` remain green before adding any stronger dungeon-gen-specific overlay coverage.

---

## Definition of done

10.8 is complete only when:

1. Bank 1 ownership is proven and asserted.
2. MMU access is gatewayed for preload/cache paths.
3. Tier cache is stable under upgraded smoke coverage.
4. Overlay cache is only enabled after the same ownership proof and runtime validation.
5. No step relies on "unused Bank 1 RAM" as an undocumented assumption.
