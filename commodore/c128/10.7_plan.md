# 10.7 — Full 80-Column UI Layout Adoption (C128)

Status: Planned  
Priority: High  
Owner: C128 track

## Objective
Replace the centered 40-column carry-over with native 80-column layouts on C128 VDC for:
- viewport framing policy,
- message lines/history,
- status panel,
- title/help/menu flows,
- related constants/tables and layout math.

This plan is implementation-ready and decision-complete for multi-agent execution.

## Why this exists
Phase 10.1 delivered VDC backend and 80-column mode, but current rendering still uses a global center offset and many 40-column constants. The result is an 80-column screen that mostly displays legacy 40-column composition in the middle.

## Non-goals
- No C64 layout changes.
- No gameplay logic rule changes.
- No bank-placement policy regressions (A8 constraints remain mandatory).

## Hard Invariants
1. C64 visuals and behavior must remain unchanged.
2. C128 must keep all A8 placement assertions and layout harness gates green.
3. No runtime `zp_machine_type` branching in `common/` for this work; use compile-time split (`#if C128`).
4. No regressions to C128 help/title stability fixes (C5) or prompt/input fixes (C3/C4/C2).

## Key Decisions (locked)
1. Viewport policy for 10.7:
   - Keep gameplay viewport width at 38 columns in this phase (low-risk), but position and surrounding UI use full 80-column composition.
   - A future task may widen gameplay viewport after LOS/render assumptions are isolated.
2. Screen primitive policy:
   - Remove implicit global centering from generic C128 cursor math.
   - Apply placement through explicit layout constants/zones.
3. Title strategy:
   - Prefer separate C128 title data/art layout path over patching the 40-col stream in place.

## Work Breakdown

### Phase A — Screen Layout Core (must land first)
Files:
- `commodore/c128/screen_vdc.s`
- (if needed) `commodore/c128/main.s` for new compile-time constants export

Changes:
1. Remove implicit `SCREEN_COL_OFFSET` behavior from `screen_set_cursor`.
2. Keep `SCREEN_COLS=80`, add explicit named zone constants for C128:
   - `MSG_X`, `MSG_W`
   - `VIEW_X`, `VIEW_W` (still 38 in 10.7)
   - `STATUS_X`, `STATUS_W`
   - `INPUT_X`, `INPUT_W`
3. Update helper routines that currently assume centered offset (`screen_flash_at`, related callers) to use explicit zone coordinates.
4. Preserve C64-compatible interfaces (`screen_put_*`) while changing C128 internal placement behavior.

Acceptance:
- C128 compiles and current tests pass with no implicit +20 offset in generic cursor path.

### Phase B — Message/Status Core Layout Migration
Files:
- `commodore/common/ui_messages.s`
- `commodore/common/ui_status.s`
- `commodore/common/zeropage.s` comments/constants if needed

Changes:
1. `ui_messages.s`:
   - Replace hardcoded `MSG_HIST_LEN = 40` with compile-time `SCREEN_COLS` (C128 path) / `40` (C64 path).
   - Update clear/init/history logic to avoid fixed-size assumptions (640 bytes on C128 vs 320).
   - Update clamp logic currently pegged to 40-column width.
2. `ui_status.s`:
   - Replace hardcoded column literals with named layout constants.
   - Define separate C128 column map that uses full width (not packed to first 40 cols).

Acceptance:
- Messages and status render correctly in full-width composition on C128.
- C64 status/message output remains bit-for-bit equivalent.

### Phase C — Overlay/UI Screen Migration
Files (minimum):
- `commodore/common/ui_help.s`
- `commodore/common/ui_help_data.s`
- `commodore/common/disk_swap.s`
- `commodore/common/title_data.s` (or new `title_data128.s`)
- `commodore/c128/main.s` (title/menu coordinates)

Files (likely follow-on in same phase):
- `commodore/common/ui_inventory.s`
- `commodore/common/ui_character.s`
- `commodore/common/ui_recall.s`
- `commodore/common/ui_home.s`
- `commodore/common/ui_store.s`

Changes:
1. Help:
   - Replace fixed border cols (`39`, `38`) with `SCREEN_COLS` math on C128.
   - Reflow title/footer/content anchors for 80-column readability.
2. Disk swap/menu prompts:
   - Replace hardcoded center constants with `((SCREEN_COLS - text_len) / 2)` compile-time expressions.
3. Title:
   - Implement C128-specific title layout/data path (preferred) or guarded coordinate expressions.
4. Menu/home/store/inventory/character/recall:
   - Convert fixed `zp_cursor_col` literals to C128 layout constants.
   - Keep C64 literals in `#else`.

Acceptance:
- Title/help/menu/home/store/inventory/character/recall all visibly use 80-column composition.

2. Ensure effects/flash/projectile overlays align with viewport under new coordinate model.

Acceptance:
- Gameplay render/effects alignment remains correct after removing global centering model.

## Multi-Agent Parallelization Plan

Agent 1 (Core):
- Phase A + D owner (screen primitives + viewport framing).

Agent 2 (Text bars):
- Phase B owner (messages/status).

Agent 3 (Screens):
- Phase C owner (help/title/menu/special UI screens).

Agent 4 (Quality/docs):
- Phase E owner (tests/harness/docs), runs last and iterates until all gates pass.

Merge order:
1. Agent 1 branch merges first.
2. Agents 2 and 3 rebase onto Agent 1 and merge.
3. Agent 4 rebases last, finalizes tests/docs.

## Verification Matrix

2. `cd commodore/c64 && make test`

Manual C128:
1. Title and menu alignment across full 80 columns.
2. Help frame/content uses full width and dismisses cleanly.
3. In-game message area fills expected width.

5. Inventory/equipment/character/recall/store/home screens visually aligned.

## Exit Criteria (Definition of Done)
1. C128 no longer depends on centered 40-col carry-over in generic screen path.
2. Primary UI surfaces (title/help/messages/status/menu) are native 80-col layouts.
3. All automated suites pass on C64 and C128.
4. New layout guards prevent accidental reintroduction of 40-col assumptions.
5. BUILDPLAN marks 10.7 done; history captures implementation and validation.
